# 桌面安装包打包方案设计

> 目标：把 SimplePlanePlatform 做成像 Clash 一样的桌面客户端 —— macOS 出 `.dmg`、
> Windows 出 `.exe/.msi`，用户**下载 → 双击安装 → 点连接**即可使用，
> 机器上**无需预装 Java / Node / Rust 任何运行时**。
>
> 技术栈选型（已与负责人确认）：**Tauri 桌面壳 + jlink 精简 JRE**，**macOS 与 Windows 双平台同等支持**。
>
> 本文是实施蓝图，分阶段落地；每个阶段均可独立验证、独立交付。

---

## 1. 问题背景：当前为什么"不能一键启动"

Clash 用 Go 编写，编译产物是**单个静态二进制、零运行时依赖**，下载即用。
本项目则是三种语言混合，每种都带运行时依赖：

| 模块 | 语言 | 运行时依赖 | 产物 |
| --- | --- | --- | --- |
| proxy-local / proxy-remote | Java 8 | 需要 JRE | fat-jar（已用 maven-shade-plugin 打好，main class `com.proxy.local.ProxyLocalServer`） |
| dashboard | Node.js | 需要 Node | `server.js` + `public/` 静态资源 |
| tun-adapter | Rust | **无**（静态二进制） | `tun-adapter` 可执行文件 |

控制链（依据 `dashboard/server.js` 实际逻辑）：

```
用户 → dashboard(Node, :3000) ──spawn('java', '-jar', proxy-local.jar)──→ proxy-local(:1080)
                              └─spawn(sudo, tun-adapter -c tun.toml)────→ tun-adapter(虚拟网卡)
```

因此小白用户当前要先装 **JRE + Node** 两套环境，极易"卡在配置环境这一步就放弃"。
**消除这两个运行时依赖，是本方案的核心。**

---

## 2. 总体方案：把所有运行时一起塞进安装包

一句话原则：**用户机器上不该需要的东西，全部打进安装包里随程序分发。**

最终安装包内部结构（概念）：

```
SimplePlanePlatform.app / 安装目录
├── 主程序（Tauri 壳，Rust 编写，负责 GUI 窗口 + 进程编排）
├── resources/
│   ├── runtime/            ← jlink 生成的精简 JRE（自带，不用系统 Java）
│   │   └── bin/java(.exe)
│   ├── proxy-local.jar     ← maven-shade fat-jar
│   ├── tun-adapter(.exe)   ← Rust 静态二进制
│   ├── webui/              ← dashboard 的 public/ 静态资源（由 Tauri webview 直接加载）
│   └── config/             ← 默认 proxy.yml / tun.toml 模板
└── （Windows 额外）wintun.dll  ← TUN 模式所需驱动
```

三个依赖各自的消除手段：

### 2.1 Java → jlink 精简 JRE（已有 fat-jar，只差裁剪运行时）

`proxy-local` 的 pom 已配置 `maven-shade-plugin`，`mvn package` 即产出可独立运行的
fat-jar（含所有依赖 + SPI 合并 + main class）。**唯一缺口是"目标机器没有 java"。**

用 JDK 自带的 **`jlink`** 生成一个只包含必要模块的迷你运行时（通常 30~50 MB，
而非整个 JDK 几百 MB），随包分发。dashboard/Tauri 启动 jar 时不再调系统 `java`，
而是调包内 `resources/runtime/bin/java`。

> Java 8 注意：jlink 是 JDK 9+ 的工具，要求代码模块化或以 classpath 方式运行。
> 实施时用 **JDK 17 的 jlink 生成运行时**，但以 `-jar`（classpath 模式）运行 Java 8 编译的
> fat-jar 完全兼容（JVM 向后兼容）。无需把业务代码改成 JPMS 模块。
> 模块清单可用 `jdeps --print-module-deps proxy-local.jar` 推导，至少包含
> `java.base, java.logging, java.naming, jdk.crypto.ec, jdk.unsupported`（Netty 需要），
> 最稳妥用 `--add-modules ALL-MODULE-PATH` 兜底再按体积优化。

### 2.2 Node → 收编进 Tauri 壳，彻底去掉 Node

dashboard 现在承担两个职责：**(a) 提供 Web UI**、**(b) 编排后台进程（spawn java / tun）**。
迁到 Tauri 后两者分别由 Tauri 原生能力替代：

- **(a) Web UI**：Tauri 用系统 WebView 显示，直接加载 `dashboard/public/` 的现有前端页面，
  **前端代码基本不用改**（只把原来 `fetch('/api/...')` 改成调用 Tauri 的 `invoke()` 命令，
  或保留一个内嵌的轻量 HTTP 层）。
- **(b) 进程编排**：`server.js` 里的 spawn/状态机/sudo 提权逻辑，用 **Tauri 的 Rust 后端 +
  sidecar 机制**重写。Tauri 的 sidecar 专门用来管理"随包分发的外部可执行文件"
  （正好对应我们的 java、tun-adapter）。

> 这样连 Node 运行时都不再需要。`server.js` 的业务逻辑（端口探测 `isPortListening`、
> sudo 错误分类 `classifySudoFailure`、状态聚合 `getSetupStatus` 等纯函数）已被单元测试覆盖，
> 迁移时可作为 Rust 重写的行为基准对照。
>
> **过渡备选**（若想先快速出包、暂不重写 server.js）：用 Node 的 SEA（Single Executable
> Application）或 `pkg` 把 `server.js` + Node 运行时打成一个二进制 `dashboard-server`，
> 作为 Tauri sidecar 挂载。改动更小，但安装包多带一份 Node 运行时（体积偏大）。
> 推荐最终走"收编进 Tauri"的彻底方案。

### 2.3 Rust（tun-adapter）→ 已是零依赖二进制，仅需交叉编译各平台

crate `tun-adapter` v0.1.0，`cargo build --release` 产出静态二进制。
各平台用各自 target 编译即可（见 §4 CI）。这是三者中最省心的。

## 3. 工具选型

| 用途 | 选型 | 理由 |
| --- | --- | --- |
| 桌面壳 + 安装包生成 | **Tauri 2.x** | Rust 编写、体积小（壳约几 MB，对比 Electron 上百 MB）、`tauri build` 原生产出 dmg/msi/nsis、sidecar 机制天然适配"随包分发外部可执行文件"、与本项目已有 Rust 调性一致 |
| Java 运行时裁剪 | **jlink**（JDK 17） | JDK 自带、生成最小化 JRE、随包分发后零系统 Java 依赖 |
| Java 应用打包 | **maven-shade-plugin**（已就位） | 现有 pom 已产出 fat-jar，无需新增 |
| Rust 跨平台编译 | **cargo + rustup target** | tun-adapter 各平台原生编译 |
| 多平台 CI 构建 | **GitHub Actions：`macos-latest` + `windows-latest`** | 桌面安装包**交叉编译极困难，标准做法是每个平台用各自 runner 原生构建**；与现有 `tests.yml` / `android-core.yml` 风格一致 |

> 为什么不用 jpackage 直接出包？jpackage 只能解决 Java 这一半，管不了 Node 和 Rust，
> 也不提供统一 GUI 壳。用 Tauri 做"总壳"把三个产物统一编排、统一出包更合适。

## 4. 多平台构建流水线（GitHub Actions）

新增 `.github/workflows/release.yml`（与现有 CI 解耦，仅在打 tag 时触发）。
关键点：**矩阵 + 各平台原生 runner**。

```yaml
# 触发：推送 v*.*.* tag 时构建并发布
name: release
on:
  push:
    tags: ["v*.*.*"]

jobs:
  build:
    strategy:
      matrix:
        include:
          - os: macos-latest    # 产出 .dmg（同时支持 arm64 + x86_64，建议分别构建或 universal）
            rust_target: ""      # 用 runner 原生架构；如需双架构再加 aarch64/x86_64-apple-darwin
          - os: windows-latest   # 产出 .msi / NSIS .exe
            rust_target: ""
    runs-on: ${{ matrix.os }}
    steps:
      - uses: actions/checkout@v4

      # 1) JDK 17（用其 jlink 生成精简 JRE；同时编译/打包 fat-jar）
      - uses: actions/setup-java@v4
        with: { distribution: temurin, java-version: "17", cache: maven }
      - name: 打包 fat-jar
        run: mvn -B -DskipTests package -pl proxy-local -am
      - name: jlink 生成精简 JRE
        run: bash scripts/build-jre.sh   # 见 §5

      # 2) Rust：编译 tun-adapter + Tauri 壳
      - uses: dtolnay/rust-toolchain@stable
      - name: 编译 tun-adapter（release）
        working-directory: tun-adapter
        run: cargo build --release

      # 3) 收集产物到 Tauri resources（jar / jre / tun-adapter / webui / wintun.dll）
      - name: 准备 Tauri 资源
        run: bash scripts/collect-resources.sh   # 见 §5

      # 4) Tauri 构建安装包
      - name: tauri build
        working-directory: desktop            # Tauri 工程目录（见 §6）
        run: npx tauri build

      # 5) 上传安装包 artifact / 发 Release
      - uses: actions/upload-artifact@v4
        with:
          name: installer-${{ matrix.os }}
          path: |
            desktop/src-tauri/target/release/bundle/**/*.dmg
            desktop/src-tauri/target/release/bundle/**/*.msi
            desktop/src-tauri/target/release/bundle/**/*.exe
```

> macOS 若要同时支持 Apple Silicon 和 Intel，需分别构建 `aarch64-apple-darwin` 与
> `x86_64-apple-darwin` 再合成 universal，或分发两个包。首版可只出 arm64。

## 5. 需要新增的构建脚本（放 `scripts/`）

实施阶段会新增以下脚本（本文档只给契约，代码在实施阶段提交）：

- `scripts/build-jre.sh` / `.ps1`：用 `jdeps` 推导模块 → `jlink --add-modules ... --output dist/runtime`
  生成精简 JRE。输入 `proxy-local/target/proxy-local-1.0.0-SNAPSHOT.jar`，输出 `dist/runtime/`。
- `scripts/collect-resources.sh` / `.ps1`：把 fat-jar、`dist/runtime`、`tun-adapter` 二进制、
  `dashboard/public`、默认配置、（Windows）`wintun.dll` 拷进 `desktop/src-tauri/resources/`。
- `scripts/build-rust.sh`（已存在，当前用于 Android）：桌面构建另走 `cargo build --release`，
  不与 Android 交叉编译逻辑混用。

## 6. Tauri 工程结构（新增 `desktop/`）

```
desktop/
├── package.json                 # 仅构建期需要 Node（开发机），最终用户不需要
├── src/                         # 前端：复用 dashboard/public 的页面与样式
│   └── index.html / app.js
└── src-tauri/
    ├── Cargo.toml
    ├── tauri.conf.json          # 声明 sidecar、resources、bundle targets(dmg/msi/nsis)、权限
    ├── resources/               # 由 collect-resources 脚本填充
    └── src/
        └── lib.rs               # Rust 后端：重写 server.js 的进程编排
            #   - start_proxy_local(): 调 resources/runtime/bin/java -jar proxy-local.jar
            #   - start_tun(): 提权拉起 tun-adapter（mac sudo / win 管理员）
            #   - status(): 端口探测 + 进程状态（对照 server.js 的 isPortListening/getStatusAll）
            #   - 通过 #[tauri::command] 暴露给前端 invoke()
```

`tauri.conf.json` 关键配置项：
- `bundle.targets`：macOS = `["dmg", "app"]`，Windows = `["msi", "nsis"]`
- `bundle.resources`：声明随包资源（runtime/jar/tun-adapter/webui/config）
- `bundle.externalBin`：声明 sidecar（java 启动器、tun-adapter）
- Windows `bundle.windows`：嵌入 `wintun.dll`、应用 manifest 声明 `requireAdministrator`

## 7. 权限与驱动（TUN 模式绕不开）

TUN 模式要创建虚拟网卡，**必然需要系统提权**（Clash 同样如此）：

### macOS
- 创建 `utun` 设备需要 root（`dashboard/server.js` 现用 `sudo -n` 临时方案）。
- 正式产品方案，按推荐度排序：
  1. **安装期配置 sudoers**：安装包在用户授权后写入一条 sudoers 规则，
     允许免密执行 tun-adapter（项目已有 `dashboard/setup-tun-permissions.sh` 思路可复用）。
  2. **首启请求授权**：首次启用 TUN 时弹系统授权框（`AuthorizationExecuteWithPrivileges`，
     已废弃但仍可用）或做 privileged helper（`SMJobBless`，最规范但工程量大）。
- 需要 **Apple Developer 签名 + 公证（notarization）**，否则 Gatekeeper 拦截。首版内测可
  用临时签名 + 引导用户"右键打开"。

### Windows
- TUN 模式依赖 **WinTUN 驱动**（`wintun.dll`，需随包分发）。
- 程序需以**管理员身份**运行：在 Tauri 的应用 manifest 声明 `requireAdministrator`，
  或安装时配置。
- 安装包建议做**代码签名**，否则 SmartScreen 警告。首版内测可接受未签名 + 用户手动放行。

> 这部分是"内部能跑"与"正式产品"的主要差距。首版可先实现"系统代理模式"（无需 TUN、
> 无需驱动/提权，仅改系统代理设置），把 TUN 模式作为进阶选项，降低首版门槛。

## 8. 分阶段实施路线图

每阶段可独立验证、独立交付，建议按序推进：

| 阶段 | 目标 | 验证标准 | 主要改动 |
| --- | --- | --- | --- |
| **一** | 消除 Java 依赖 | 在**卸载了系统 Java** 的机器上，用包内精简 JRE 能启动 proxy-local | `scripts/build-jre.sh`；dashboard `getJarPath` 旁加"优先用包内 JRE"分支 |
| **二** | 收编 Node（或过渡 SEA） | 不装 Node 能拉起控制面 | Tauri Rust 后端重写编排 / 或 Node SEA 打包 |
| **三** | Tauri 壳 + 双平台出包 | mac 出 .dmg、win 出 .msi，本机双击安装即用（系统代理模式） | 新增 `desktop/` 工程、`collect-resources` 脚本 |
| **四** | TUN 模式权限/驱动 | mac 授权拉起 utun、win 装 WinTUN 后 TUN 可用 | sudoers/helper、wintun.dll、manifest |
| **五** | CI 发布流水线 | 打 tag 自动产出双平台安装包并发 Release | `.github/workflows/release.yml` |

## 9. 风险与取舍

- **体积**：精简 JRE（~40MB）+ tun-adapter + webui，最终安装包预计 60~120MB（Clash 因纯 Go 仅
  ~20MB）。这是混合技术栈的固有代价；彻底瘦身需把 Java 数据面逐步迁到 Rust（`plane-core`
  已是 Rust，存在长期收敛到单语言的可能，但属架构级演进，不在本方案范围）。
- **签名/公证成本**：正式分发需 Apple 开发者账号（公证）与 Windows 代码签名证书，均有年费；
  内测阶段可用未签名包 + 引导用户放行。
- **维护成本**：三套产物 + 双平台 = 6 类构建路径，必须靠 CI 固化，避免手工打包出错。
- **server.js 逻辑迁移风险**：进程编排、sudo 提权、状态机迁到 Rust 需谨慎；好在核心纯函数
  已有单元测试（`dashboard/test/`）作为行为基准。

---

## 10. 结论

技术上完全可行，路径清晰：**Tauri 做总壳统一编排，jlink 干掉 Java 依赖，Node 收编进 Tauri 壳，
Rust 二进制原生编译，GitHub Actions 双平台 runner 出包。** 最终交付 mac `.dmg` + Windows
`.msi/.exe`，达成"下载即用、零环境配置"的 Clash 式体验。

唯一无法回避的是 **TUN 模式的提权与驱动**（任何同类工具都一样），建议首版先以**系统代理模式**
降低门槛，TUN 作为进阶能力分阶段补齐。

实施按 §8 路线图分五阶段推进，每阶段独立可验证。
