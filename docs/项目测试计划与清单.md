## 一、项目概览与被测对象建模

> 用例完整度的地基是"建模"——先把系统拆清楚，才知道该测哪些地方。

### 1.1 系统架构

SimplePlanePlatform 是一套自建的代理 / TUN 网络工具，核心数据流：

```
                          ┌─────────── 系统代理模式 ───────────┐
应用流量 ──► proxy-local ──┤                                    ├──► proxy-remote ──► 公网
            (Java :1080)  └─ 加密(ChaCha20/AES) ─ Netty 传输 ──┘    (Java :9090)

                          ┌─────────── TUN 模式 ───────────────┐
应用流量 ──► utun9 ──► tun-adapter ──► SOCKS5 :1080 ─► proxy-local ──► proxy-remote ──► 公网
            (路由劫持)   (Rust:协议栈/FakeDNS/路由)
```

### 1.2 模块清单与职责

| 模块 | 语言 | 职责 | 测试关注重点 |
|------|------|------|-------------|
| **proxy-crypto** | Java | 加解密（ChaCha20-Poly1305 / AES-GCM / AES-CTR-HMAC） | 安全核心，加解密往返、篡改检测、错误密钥 |
| **proxy-common** | Java | 公共组件（传输、流控 FlowPermit 等） | 流控逻辑、背压 |
| **proxy-transport-netty** | Java | 基于 Netty 的传输层 | 编解码、连接管理 |
| **proxy-local** | Java | 本地代理（SOCKS5/系统代理，:1080） | 转发逻辑、系统代理开关、DNS |
| **proxy-remote** | Java | 远端代理（:9090） | 出站转发、与 local 的协议互通 |
| **proxy-exchange** | Java | 数据交换/协议 | 协议编解码 |
| **proxy-cluster** | Java | 集群/多节点 | 节点选择、故障转移 |
| **tun-adapter** | Rust | TUN 设备、用户态协议栈、FakeDNS、路由、SOCKS5 客户端 | 路由决策、FakeDNS、SOCKS5、DNS 备份恢复 |
| **plane-core** | Rust | Android 数据面（交叉编译为 .so） | 核心转发逻辑 |
| **dashboard** | Node.js | Web 管理面板（进程管理、状态、一键启停） | 进程编排、sudo 处理、错误分类 |
| **android-app** | Kotlin | Android 客户端 | UI、VPN Service |

### 1.3 核心价值场景（必须保障的主链路）

1. **系统代理模式翻墙**：proxy-local + proxy-remote 启动 → 配置系统代理 → 能访问公网
2. **TUN 模式翻墙**：tun-adapter + proxy-local + proxy-remote 启动 → 全局流量劫持 → 能访问公网，且内网正常
3. **加密链路**：数据经"加密→传输→解密"全链路后内容不变、被篡改能检测
4. **安全退出**：TUN 退出后路由、DNS 能完整恢复，不留下断网状态
5. **Dashboard 一键启停**：从面板能正确编排启动顺序、并准确反映真实状态

## 二、测试目标与质量度量

### 2.1 测试目标

| 项 | 内容 |
|----|------|
| 验证目标 | 保障 5 条核心价值场景在正常与异常情况下均行为正确 |
| 非目标（本轮不测） | Android 客户端 UI 自动化、大规模集群压测、跨地域网络 |
| 质量底线 | 核心加密链路、TUN 路由、DNS 恢复必须有自动化测试覆盖且全过 |

### 2.2 质量度量指标

| 指标 | 目标值 | 统计方式 |
|------|--------|---------|
| 核心模块行覆盖率（proxy-crypto / tun-adapter） | ≥ 80% | JaCoCo / cargo-tarpaulin |
| 整体行覆盖率 | ≥ 60% | 覆盖率工具汇总 |
| 单元测试通过率 | 100% | CI 统计 |
| 核心主链路 E2E | 至少 1 条自动化跑通 | 脚本断言 |
| 每个历史 bug 回归测试 | 100% 覆盖已修 bug | 见 §6 回归测试集 |

## 三、测试分层策略（本项目的测试金字塔）

```
              ╱╲
             ╱E2E╲          系统代理/TUN 翻墙整链路脚本   （少，约 5 条）
            ╱──────╲
           ╱ 集成测试 ╲      local↔remote 转发、加密链路、 （约 15~20 条）
          ╱           ╲     TUN 路由集成
         ╱──────────────╲
        ╱    单元测试      ╲  加密、路由、FakeDNS、协议    （主力，目标 100+）
       ╱__________________╲  编解码、配置解析、流控、dashboard纯函数
```

| 层级 | 占比 | 本项目对应内容 | 工具 |
|------|------|--------------|------|
| 单元 | ~70% | 加解密、路由规则、FakeDNS、SOCKS5 解析、配置解析、流控、dashboard 纯逻辑函数 | JUnit + Mockito（Java）/ `#[test]`（Rust）/ node:test（Node） |
| 集成 | ~20% | local↔remote 真实转发、加密全链路、TUN 路由集成 | JUnit 集成 / Rust `tests/` / 脚本 |
| E2E | ~10% | 完整翻墙链路、TUN 启停与 DNS 恢复 | bash 脚本 + curl 断言 |

### 4.1 proxy-crypto（加密核心）⭐ 最高优先级

> 加密是安全核心，又是纯函数易测，性价比最高。当前仅 1 个测试（CryptoVectorGenTest），远远不够。

## 八、自动化与 CI 落地路线

### 8.1 工具选型

| 模块 | 测试框架 | 覆盖率工具 |
|------|---------|-----------|
| Java（各 proxy-* 模块） | JUnit 5 + Mockito | JaCoCo |
| Rust（tun-adapter / plane-core） | 内置 `#[test]` + `tests/` | cargo-tarpaulin |
| Node（dashboard） | node:test（零依赖） | c8 / built-in coverage |
| E2E | bash + curl + 断言 | — |

### 8.2 CI 流水线（建议接入 GitHub Actions）

```
push / PR
   │
   ├─ Java:  mvn test + jacoco:report        （门控：核心模块覆盖率 ≥ 80%）
   ├─ Rust:  cargo test + cargo clippy        （门控：全过 + 无 warning）
   ├─ Node:  node --test dashboard/           （门控：全过）
   │
   └─ 全绿 → 允许合并；有红 → 阻断
```

### 8.3 准入 / 准出标准

**准入（提测）**：各模块可构建、单元测试通过、被测对象建模完成。

**准出（发布）**：
- [ ] 单元测试 100% 通过
- [ ] 核心模块（crypto / tun-adapter）覆盖率 ≥ 80%
- [ ] §1.3 五条核心主链路 E2E 跑通
- [ ] §6 回归测试集全部补齐并通过
- [ ] 安全专项（§5.1）无高危

## 九、落地建议（按性价比排序）

如果时间有限，建议按这个顺序推进，每一步都能立竿见影地提升"测试成熟度"：

1. **接 CI + 覆盖率**（半天）——让 push 自动跑现有测试并出覆盖率数字，立刻有可量化的成熟度信号。
2. **补 proxy-crypto 单元测试**（§4.1）——安全核心、纯函数易测、12 条用例性价比最高。
3. **补 §6 回归测试集**——把已修的 4 个 bug 变成永久护栏，最能体现"懂测试驱动质量"。
4. **补 tun-adapter 路由/SOCKS5 单元测试**（§4.2）——在已有良好基础上补强。
5. **写 local↔remote 集成测试 + 加密链路**（§4.3）——验证主链路。
6. **dashboard 纯函数单元测试**（§4.5）——从 0 到 1，覆盖率提升明显。
7. **E2E 脚本化**——把手动验证"启动→curl→断言"自动化。
