## 1. 背景与问题陈述

### 1.1 现状

SimplePlanePlatform 已具备完整代理能力，但由三种语言混合构成，各带运行时依赖：

| 模块 | 语言 | 运行时依赖 |
| --- | --- | --- |
| proxy-local / proxy-remote | Java 8 | 需要系统安装 JRE |
| dashboard（控制面板 + Web UI） | Node.js | 需要系统安装 Node |
| tun-adapter | Rust | 无（静态二进制） |

启动控制链（依据 `dashboard/server.js`）：`dashboard` 用 `spawn('java', '-jar', proxy-local.jar)`
拉起代理、用 `spawn(sudo, tun-adapter)` 拉起 TUN。

### 1.2 问题

普通用户（"小登"）想使用必须**自行安装并配置 Java + Node 两套环境**，极易在"配置环境"
这一步放弃。对比 Clash（Go 单二进制，下载即用），本项目使用门槛过高。

### 1.3 目标用户与场景

- **目标用户**：无开发背景的普通终端用户。
- **核心场景**：在官网/分发渠道下载安装包 → 双击安装 → 打开窗口 → 点"连接" → 正常上网。
  全程不接触命令行、不安装任何运行时。

## 2. 需求范围

### 2.1 范围内（In Scope）

- macOS 与 Windows 双平台桌面安装包（`.dmg` / `.msi` 或 `.exe`）。
- 安装包自带全部运行时（Java 精简 JRE、Web UI、Rust 二进制），目标机器零运行时依赖。
- 图形化客户端：连接/断开、状态显示、基本配置。
- 系统代理模式（首版主推，无需提权/驱动）。
- TUN 透明代理模式（进阶，分阶段交付，涉及提权/驱动）。
- CI 自动化产出双平台安装包。

### 2.2 范围外（Out of Scope）

- Linux 桌面安装包（后续可扩展，本期不做）。
- Android 客户端（已有独立方案 `docs/design/android-client-plan.md`）。
- 把 Java 数据面重写为 Rust（架构级演进，非本期目标）。
- 应用商店上架（Mac App Store / Microsoft Store）。

## 3. 功能需求（Functional Requirements）

> EARS 句式约定：
> - 普遍型：`系统应 <响应>`
> - 事件驱动：`当 <触发> 时，系统应 <响应>`
> - 状态驱动：`在 <状态> 期间，系统应 <响应>`
> - 条件型：`若 <条件>，则系统应 <响应>`

### REQ-001 零运行时依赖运行（P0）
**作为** 普通用户，**我希望** 在没有预装 Java/Node 的机器上也能运行本软件，**以便** 不必配置开发环境。

验收准则：
1. 在一台**未安装任何 JRE/JDK** 的机器上，安装本软件后启动，代理核心（proxy-local）应能正常运行。
2. 在一台**未安装 Node.js** 的机器上，安装本软件后启动，控制界面应能正常显示与操作。
3. 当软件启动时，系统应使用**安装包内自带的运行时**，而不应依赖或读取系统已安装的 Java/Node。

### REQ-002 双击安装、图形化启动（P0）
**作为** 普通用户，**我希望** 下载后双击即可安装并出现窗口，**以便** 无需任何命令行操作。

验收准则：
1. macOS 用户双击 `.dmg` 拖入应用程序后，从启动台/应用程序打开应出现客户端窗口。
2. Windows 用户双击安装包完成安装后，从开始菜单/桌面快捷方式打开应出现客户端窗口。
3. 当客户端窗口打开时，系统应展示连接开关与当前连接状态，全程无需用户接触终端。

### REQ-003 一键连接/断开（P0）
**作为** 普通用户，**我希望** 点一个按钮就能连接或断开代理。

验收准则：
1. 当用户点击"连接"时，系统应依序拉起代理核心并在就绪后将状态显示为"已连接"。
2. 当用户点击"断开"时，系统应停止代理相关进程并将状态显示为"已断开"。
3. 在连接过程中，系统应展示进行中的过渡状态（如"连接中…"），不应让界面无反馈。

### REQ-004 系统代理模式（P0）
**作为** 普通用户，**我希望** 首版能用最简单、无需授权的方式代理上网。

验收准则：
1. 当用户在系统代理模式下点击"连接"时，系统应自动设置操作系统代理指向本地代理端口。
2. 当用户断开或退出软件时，系统应自动恢复系统代理设置到先前状态。
3. 系统代理模式运行时，系统**不应**要求管理员/ root 权限，也不应安装任何驱动。

### REQ-005 TUN 透明代理模式（P1，进阶）
**作为** 进阶用户，**我希望** 可选启用 TUN 模式实现全局透明代理。

验收准则：
1. 当用户首次启用 TUN 模式时，系统应通过操作系统标准机制请求一次必要的提权授权。
2. 在 macOS 上启用 TUN 时，系统应能创建 utun 设备；在 Windows 上，系统应使用随包分发的 WinTUN 驱动。
3. 若用户拒绝授权或驱动不可用，则系统应给出明确提示并保持在系统代理模式可用状态，不应崩溃。

### REQ-006 进程编排与状态可见（P0）
**作为** 用户，**我希望** 清楚地看到代理是否真的在运行。

验收准则：
1. 系统应探测本地代理端口监听状态，并据此显示真实运行状态（对照现有 `isPortListening` 行为）。
2. 当后台进程异常退出时，系统应将状态更新为"未运行/异常"，不应继续显示"已连接"。
3. 当用户退出软件时，系统应清理其拉起的所有后台子进程，不应残留僵尸进程。

### REQ-007 基础配置（P1）
**作为** 用户，**我希望** 能填入远端节点地址/密钥等基本配置。

验收准则：
1. 系统应提供界面让用户查看与修改远端地址、端口、加密密钥等基础参数。
2. 当用户保存配置时，系统应将其写入用户可写的配置目录，重启后仍生效。
3. 系统应内置一份可直接连接的默认/示例配置模板。

## 4. 非功能需求（Non-Functional Requirements）

### REQ-NFR-001 多平台一致性（P0）
系统应在 macOS（至少 Apple Silicon）与 Windows（x64）上提供功能等价的安装包与体验。

### REQ-NFR-002 安装包体积可接受（P1）
单平台安装包体积应控制在合理范围（目标 ≤ 150MB；精简 JRE 预计 ~40MB）。
> 说明：混合技术栈天然大于 Clash（纯 Go ~20MB），此为已知取舍。

### REQ-NFR-003 可自动化构建（P0）
系统应能通过 CI 在各自平台 runner 上自动产出安装包，不依赖手工打包步骤。

### REQ-NFR-004 安全分发（P1）
分发包应支持代码签名（macOS 公证 / Windows 代码签名）；内测阶段允许未签名但需给出放行引导。

### REQ-NFR-005 行为不回归（P0）
进程编排逻辑从现有 `dashboard/server.js` 迁移时，端口探测、sudo 错误分类、状态聚合等
核心行为不应回归（以 `dashboard/test/` 既有单元测试为基准）。

## 5. 约束与假设

- **C-1**：proxy-local 已用 maven-shade-plugin 产出可独立运行 fat-jar（main class `com.proxy.local.ProxyLocalServer`），本期复用，不改打包方式。
- **C-2**：tun-adapter 为零依赖 Rust 二进制，各平台原生编译。
- **C-3**：桌面安装包**不做跨平台交叉编译**，各平台用各自 CI runner 原生构建。
- **C-4**：jlink 用 JDK 17 运行时生成，以 classpath 模式运行 Java 8 fat-jar（向后兼容），不要求把业务代码 JPMS 模块化。
- **A-1**：TUN 模式的提权与驱动是同类工具的固有要求（Clash 亦然），用户可接受一次性授权。

---

## 6. 验收总览（Definition of Done）

本特性达成"完成"的总判据：

1. 在干净机器（无 Java/无 Node）上，双击安装包安装后可启动并在系统代理模式下正常上网（覆盖 REQ-001/002/003/004/006）。
2. macOS 与 Windows 均有可用安装包，体验等价（REQ-NFR-001）。
3. CI 打 tag 可自动产出双平台安装包（REQ-NFR-003）。
4. TUN 模式在两平台可分阶段验证通过（REQ-005）。
5. 迁移后的进程编排行为通过既有单元测试基准（REQ-NFR-005）。
